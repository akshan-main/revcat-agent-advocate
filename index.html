What is this project?

This is a job application for a specific role at RevenueCat: the Agentic AI Developer & Growth Advocate position ($10K/month, 6-month contract). Instead of submitting a resume, the idea is to submit a working system that already does the job.

The system is a Python CLI tool that does several things a developer advocate would normally do by hand:

  • Reads RevenueCat's documentation and builds a searchable index
  • Writes technical blog posts with citations back to the docs
  • Runs growth experiments (like generating SEO pages)
  • Files structured product feedback based on doc analysis
  • Drafts tweets, GitHub issue responses, and Reddit replies
  • Publishes a static website showing all of this work
  • Logs every action to a tamper-evident ledger (hash chain)

The output is a static website hosted on GitHub Pages. A reviewer at RevenueCat can visit the URL, click through the content, check the ledger, and verify that everything traces back to real documentation.

What is RevenueCat?

RevenueCat is a company that handles in-app subscriptions for mobile apps. If you have an iOS or Android app that charges users a monthly fee, RevenueCat manages the billing, receipts, entitlements, and analytics. They provide SDKs, a REST API, a dashboard, and recently an MCP server (a protocol that lets AI agents call their API).

They publish their documentation at revenuecat.com/docs and notably provide:

  • An LLM-optimized text index of all their docs (a single text file listing every page)
  • .md mirrors of every doc page (append .md to any doc URL to get markdown)
  • An MCP server at mcp.revenuecat.ai/mcp with 26 tools
  • A Charts API for querying metrics like MRR, churn, LTV

Why build this instead of a resume?

The job posting asks for someone who can operate autonomously, create developer content, run growth experiments, and provide product feedback. This system does all of those things, and by submitting it, the application is the proof of work.

The hash-chained ledger is the differentiator. Every action the system takes (ingesting docs, writing a post, running an experiment) creates a log entry. Each entry's hash depends on the previous entry's hash. If anyone tampers with an entry, the chain breaks. This means a reviewer can verify that the outputs are real and haven't been faked after the fact.

How it works, step by step

1. Ingest documentation

The system downloads RevenueCat's LLM docs index, then fetches the .md version of each documentation page. It stores these locally and computes a SHA256 hash of each page. This creates a snapshot; if a doc page changes later, we know.

It then builds two search indexes over the cached docs:

  • BM25 keyword search: traditional term-frequency scoring
  • RAG (retrieval-augmented generation): splits docs into chunks, embeds them via Hugging Face Inference API (all-mpnet-base-v2, 768-dim), stores in ChromaDB Cloud, and uses a cross-encoder reranker (ms-marco-MiniLM-L-12-v2) via HF API at query time. Everything runs remotely, no local models, no local vector storage. Final scoring is 70% semantic + 30% BM25.

2. Write content

Given a topic, the system:

  1. Searches the doc index for relevant pages
  2. Builds an outline (using Claude API if available, otherwise a template)
  3. Generates a draft with citations in [Source](url) format
  4. Verifies every citation link is reachable (HEAD request)
  5. Checks that doc SHA256 hashes still match
  6. Saves to the database and outputs markdown

Without an Anthropic API key, it uses Jinja2 templates to assemble content from doc snippets. Less natural, but still includes real citations.

3. Run experiments

There are 4 predefined growth experiments:

ExperimentWhat it does
programmatic-seoGenerates SEO pages targeting keywords like "RevenueCat vs Adapty"
content-seriesWrites a connected series of tutorials on one feature area
community-blitzDrafts responses to unanswered RevenueCat questions
integration-showcaseWrites integration guides for Flutter, React Native, etc.

Each experiment records a hypothesis, a metric, and results. The SEO experiment actually generates pages; the others scaffold the structure.

4. Generate feedback

The system analyzes the cached docs and identifies potential issues: missing information, inconsistencies, unclear explanations. If Claude API is available, it uses an LLM to find these; otherwise it runs heuristic checks (broken internal links, missing code examples, etc.).

Each feedback item has: title, severity (critical/major/minor/suggestion), area (sdk/api/docs/etc.), reproduction steps, expected behavior, actual behavior, and a proposed fix.

There's also a repro harness that runs scripted API scenarios against RevenueCat's API (or mocks in demo mode) and detects friction: slow responses, unhelpful error messages, missing fields, undocumented error codes. These produce golden test snapshots that can catch regressions.

5. Social engagement

Three modules scan public platforms for RevenueCat-related discussions:

  • GitHub: scans 6 RC repos (purchases-ios, android, flutter, unity, kmp, js) for open issues
  • Reddit: searches 6 subreddits (iOSProgramming, androiddev, FlutterDev, etc.) for RC-related posts
  • Twitter/X: drafts tweets using doc context

All responses are drafts only by default. Nothing posts externally unless DRY_RUN=false and platform credentials are configured.

6. Build the website

The build-site command takes everything from the database (Turso cloud SQLite) and generates a static HTML site using Jinja2 templates. Pages:

PageShows
/applyApplication letter + proof links + quickstart for reviewers
/contentList of all generated posts, each with citations and verification status
/experimentsGrowth experiments with hypothesis and results
/feedbackProduct feedback items with severity and repro steps
/ledgerHash-chained log of every action, with chain verification status
/roiOutput dashboard: counts of everything produced
/runbookOperator reference: all commands, safety gates, architecture

7. The ledger

Every command creates a row in the run_log table (Turso cloud). Each row contains:

  • What command was run and when
  • What inputs it received
  • What doc sources it used (with SHA256 hashes)
  • What tool calls it made (e.g., Claude API calls)
  • What it produced
  • A hash: sha256(previous_entry_hash + this_entry_as_json)

Because each hash depends on the previous one, changing any entry retroactively would break the chain from that point forward. verify-ledger checks the whole chain.

Optionally, if LEDGER_HMAC_KEY is set, each entry also gets an HMAC-SHA256 signature.

Safety model

The system has three safety flags, all defaulting to the safe option:

FlagDefaultWhat it prevents
DRY_RUNtrueBlocks posting to Twitter, GitHub, Reddit. All responses stay as drafts.
ALLOW_WRITESfalseBlocks POST/PUT/DELETE requests to RevenueCat's API. Only GET allowed.
DEMO_MODEfalseUses mock API responses instead of real ones. All output labeled [DEMO].

There's also a red-team governance suite: 16 adversarial test cases across 5 categories that test whether the response-drafting system can be tricked into:

  • Leaking system prompts
  • Disclosing internal data or PII
  • Engaging with hostile threads
  • Bashing competitors
  • Making false claims about RevenueCat

These run as part of the regular test suite (7 failure detectors, each tested).

What this does NOT do

Being honest about the boundaries:

  • Without an Anthropic API key, content generation uses templates, readable but not natural prose. The Claude API makes everything significantly better.
  • Without RevenueCat credentials, demo mode uses mock API responses. The repro harness, charts, and customer queries only work with real credentials.
  • The social modules don't actually post anything in default config. They draft responses. Posting requires explicit opt-in (DRY_RUN=false) plus platform credentials.
  • The SEO pages are generated locally, not deployed to any search engine. They demonstrate the capability but don't have real traffic.
  • The competitive intelligence only reads public pricing pages and blogs. No scraping behind auth, no reverse engineering. If a page returns an error, that competitor is skipped.
  • The RAG pipeline uses remote models via Hugging Face Inference API. Quality depends on the embedding model and doc coverage. It's not magic; it's term matching with vector similarity.
  • The citation verification checks that links are reachable and doc hashes match. It does not verify that the content actually supports the claim; that requires human review.
  • Growth experiment "results" in demo mode are structural: they show that pages were generated and metrics were collected, not that real traffic or conversions occurred.

How to run it

Quickest path (demo mode, no API keys needed except Anthropic for good content):

pip install -e ".[dev]"
export DEMO_MODE=true
export ANTHROPIC_API_KEY=sk-ant-...    # optional but makes content much better
revcat-advocate demo-run
python -m http.server -d site_output 8000
# open http://localhost:8000

Run the tests:

pip install -e ".[dev]"
pytest tests/ -v    # 259 tests, ~35 seconds
